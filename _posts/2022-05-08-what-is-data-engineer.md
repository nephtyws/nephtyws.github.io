> 혹시 코딩 할 줄 아세요?

2019년에 이직했을 당시 회사의 개발자 동료에게 들어본 말입니다. 당시 저는 적당한 규모의 스타트업의 첫 번째 데이터 엔지니어로 입사했었는데, 아무래도 데이터 엔지니어라는 포지션이 생소하고 어떤 일을 하는지 잘 알려져 있지 않아서 그런 것 같다고 생각했지만..

3년이 흐른 지금도 데이터 엔지니어라는 포지션의 정의, 그리고 도대체 회사에서 어떤 일을 하는지에 대한 것들이 다른 직군 분들이 보시기에 정확히 정리되진 않은 것 같습니다. (심지어는 데이터 엔지니어들 사이에서도요!) 최근에 블라인드 게시글 중 `데이터 엔지니어를 개발자로 취급해야 하나요?` 라는 글도 봤었고, 하물며 저한테 `데이터 엔지니어는 무슨 일을 하나요?` 하고 물어봐도 한 문장으로 설명하기가 어렵기도 하고요. 그래서 이 기회에 요즘 세상의 데이터 엔지니어란 무엇이고, 어떤 일을 하며, 최근의 트렌드가 무엇인지 한 번 정리해보는 시간을 가져보려 합니다.

순서대로 쏘카, 몰로코, 토스증권, 카카오스타일의 데이터 엔지니어 채용 공고인데요. 채용 공고만 보고 감이 오시나요? 공통적으로 등장하는 항목들을 조금 정리해보면,
- 서비스의 다양한 곳에서 발생하는 정형/비정형 데이터를 수집하기 위한 배치/실시간 파이프라인 구축 및 운영
- 데이터 레이크/데이터 웨어하우스 등 전사 인원들의 원활한 데이터 분석을 위한 데이터 플랫폼 운영
- 대용량의 데이터 가공을 통해 실제 서비스 혹은 분석에 사용될 수 있는 데이터 마트 구축
- 데이터 기반 서비스 제공 및 서빙 - 관련 `*Ops` 또한 담당

이런 내용들이 공통적으로 등장하는 것을 확인할 수 있습니다. 각 내용들에 대해서 제가 데이터 엔지니어로 일할 때의 경험을 떠올려서 약간 더 살을 붙여 설명해보도록 하겠습니다.

#### 데이터 엔지니어의 업무
- **서비스의 다양한 곳에서 발생하는 정형/비정형 데이터를 수집하기 위한 배치/실시간 파이프라인 구축 및 운영**
  - 웹, 앱 등의 클라이언트 환경, DB나 서버같은 내부 백엔드 환경, 그리고 third-party tool (주로 마케팅 툴 - Facebook Ad, Appsflyer, ...) 에서 발생하는 데이터들을 ETL/ELT
    - 이를 위해 batch/streaming pipeline을 구성하고, data schema를 정의하고, ETL/ELT를 위해 분산 컴퓨팅 프레임워크 (Spark, Presto, Athena, BigQuery) 를 사용합니다.
- **데이터 레이크/데이터 웨어하우스 등 전사 인원들의 원활한 데이터 분석을 위한 데이터 플랫폼 운영**
  - 클라우드를 활용하지 않는 환경에서는 Hadoop + ecosystem (Spark, Hive, Presto 등) 을 주로 사용하고, 클라우드를 활용하는 환경에서는 AWS EMR, Athena, Glue, GCP BigQuery, Dataproc 등의 managed data service를 활용해서 데이터 플랫폼을 구축합니다.
  - Snowflake, Databricks 등의 상용 SaaS 플레이어들도 존재하고 있습니다.
- **대용량의 데이터 가공을 통해 실제 서비스 혹은 분석에 사용될 수 있는 데이터 마트 구축**
  - 전사 고객들이 데이터에 원활하게 접근할 수 있어야 하기 때문에, 원천 데이터에서 다양한 추상화를 거쳐 주로 데이터 마트라고 부르는 가공 데이터를 만듭니다.
    - 데이터 마트에는 매출 통계와 같은 단순 분석용 데이터부터 결제/정산/쿠폰/멤버십과 같은 고객 데이터, 추천 모델 생성에 사용되는 데이터, 추천 모델이 만들어낸 데이터 등 실제 고객들에게 전달되는 데이터들 또한 존재할 수 있습니다.
  - 또한 조회를 위한 BI tool (Tableau, Redash, Superset, Jupyter, Zeppelin, ...) 도 운영하며, 요새는 누구나 ad-hoc하게 분석할 수 있게 해주는 Amplitude와 같은 tool도 운영하곤 합니다.
- **데이터 기반 서비스 제공 및 서빙 - 관련 `*Ops` 또한 담당**
  - 위에 언급된 업무를 하기 위해서는 인프라를 운영하는 것이 필수적인데, 보통은 해당 조직의 데이터 엔지니어들이 데이터 관련 서비스의 DevOps (배포, 모니터링 등) 를 주된 role로 가져가는 경우가 많습니다.
    - DevOps 엔지니어분들은 항상 바쁘시기도 하시고, 데이터 환경 자체가 production 환경과는 격리된 경우도 있고, 또한 데이터 관련 배포는 실제 서비스의 배포와는 다른 주기로 이뤄지는 경우가 많기 때문입니다.
  - 또한 AB 테스트, 추천 결과 서빙 등 데이터 기반 서비스를 production으로 내보내기도 하고, DataOps, MLOps 같은 것들이 부상하면서 관련된 일도 필요하다면 데이터 엔지니어들이 같이 수행하기도 합니다.

위와 같이 데이터 엔지니어의 업무를 어느정도 정리해보았는데요. 그렇다면 이러한 업무를 수행하기 위해서 갖춰야 할 능력들은 무엇이 있을까요? 저의 주관적인 경험을 바탕으로 데이터 엔지니어가 갖춰야 할, 혹은 갖추면 좋을 법한 기술들에 대해 정리해보았습니다.

#### 데이터 엔지니어가 갖춰야 할 기술
- Tech skills
  - 분산 처리에 대한 이해
    - 데이터 엔지니어들이 주로 사용하는 고수준 프레임워크인 Spark, Athena (Presto), BigQuery, Hive 등은 전부 분산 컴퓨팅 엔진이고, 만약 분산 컴퓨팅 엔진을 사용하지 않더라도 데이터의 규모가 늘어나면 자연스럽게 분산/병렬 처리를 해야만 데이터를 원활하게 처리할 수 있기 때문에, 기본적으로 이런 분산 컴퓨팅 엔진들이 어떻게 작동하는지 알면 좋은 점이 생각보다 많습니다. 
      - 데이터가 매우 많은 상황에서는 프레임워크가 제공하는 연산 하나, 코드 한 줄이 미치는 영향이 생각보다 클 수 있기 때문입니다. 실제로 코드 몇 줄을 바꿔주는 것으로 컴퓨팅 용량을 아주 많이 차지하는 작업을 크게 개선할 수 있기도 하고요. (예를 들어 `groupByKey` 같은 연산을 다른 연산으로 바꿀 수 있다면...)
  - DB/SQL에 대한 이해
    - 데이터 엔지니어가 뛰노는 환경들 (Athena, BigQuery 등) 은 일종의 데이터베이스입니다. 일반적으로 서비스에 활용되는 RDBMS와 다른 부분도 많고, 공유하는 부분도 많습니다. 그렇기에 DB에 대한 일반적인 상식을 많이 알면 알수록, 특히 위에서 언급한 직접 운용하고 있는 데이터 플랫폼의 기반을 알면 알수록 production 환경에서 플랫폼을 운영하기 수월해집니다.
    - 또한 요즘에는 정말 많은 경우에 데이터 조회, 심지어는 삽입/삭제까지 SQL을 이용해서 하기 때문에 SQL을 잘 짤 수 있어야하며, 더 나아가 각 쿼리가 어떻게 동작하는지 아는 것도 데이터 엔지니어에겐 다다익선인 것 같습니다.
  - 프로그래밍
    - 당연하게도 데이터 엔지니어가 여러 업무를 하려면 최소 한 개 이상의 프로그래밍 언어를 아주 잘 알아야하며, 여러 도구를 운용하려면 도구에 맞는 언어를 다룰 수 있으면 좋습니다.
      - 예를 들어 Airflow와 같은 batch 작업 운용은 Python으로
      - Spark와 같은 분산 처리 프레임워크 활용은 Scala로
      - 밥먹듯이 하는 데이터 조회는 SQL로
      - production 데이터 서빙할 때는 목적에 맞는 API 서버를 만들 수 있는 Go, Javascript 등
  - (클라우드) 인프라에 대한 이해
    - 데이터 플랫폼을 전담하는 DevOps 엔지니어를 채용하는 회사는 잘 없습니다. 그렇기에 데이터 엔지니어들은 인프라 관리, 혹은 클라우드를 잘 활용해야하며, 인프라 운영과 관련된 지식들을 많이 익히고 실무에서 활용해야 하는 경우가 많습니다.
      - 클라우드 managed service 운영, 데이터 서비스용 CI/CD 파이프라인 구축, Kubernetes 운영, 모니터링 등
      - 데이터 엔지니어가 DevOps에 대한 전문가가 될 필요는 없지만, 적어도 DevOps 엔지니어 분들과의 소통을 통해 관리하는 인프라 문제를 스스로 해결할 수 있을 정도는 되어야 합니다.
- **(특히 스타트업에서 정말 중요한) Soft skills**
  - 서비스에 대한 이해
    - 전사 조직이 어떤 방향으로 나아가야 할지를 데이터 엔지니어가 제시할 수는 없지만, 일을 함께하는 동료들과 함께 서비스 혹은 BM을 개선하기 위해 어떤 데이터를 보아야할지 의견을 모으고, 적절한 지표가 설정되면 해당하는 데이터를 잘 볼 수 있도록 전사적인 흐름을 만들어내야 합니다.
      - 특히 이는 작은 규모의 스타트업, 혹은 데이터 문화가 잘 설정되지 않은 회사에서는 특히 더 필요합니다. 이런 문화가 제대로 만들어지지 않으면 데이터를 제대로 볼 수 없을 수 있고, 혹은 오히려 데이터를 잘못 봄으로써 잘못된 의사결정을 만들어내는 단초가 될 수 있습니다.
    - 혹은 이러한 과정에 데이터 엔지니어가 개입되지 않더라도, 적어도 전사적인 차원에서 어떤 데이터를 수집해야하고, 장기적인 관점에서 어떤 데이터가 더 수집되면 현재 가지고 있는 데이터들이 더 잘 활용될 수 있는지 끊임없이 고민해야합니다.
      - 그렇지 않으면 분명 데이터를 수집한지 몇 개월이 지나서 `이때 이런 데이터를 추가로 수집했었더라면 좋았을 텐데..` (데이터를 수집할 수 있는 시기를 놓치면 다시는 그 데이터를 얻지 못할 수 있습니다) 혹은 `그때 schema를 잘 정의했더라면 이런 문제가 생기지 않았을 텐데..` 하는 후회가 밀려올 수 있습니다.
  - 뛰어난 커뮤니케이션 스킬
    - 전사적으로 데이터 엔지니어가 무엇을 해야하는지 잘 정의되어있고, 데이터 엔지니어가 왜 필요한지 알며, 데이터 엔지니어가 어떠한 문제를 해결해줄지 잘 알고 있는 조직은 정말로 많지 않습니다. 혹은 전사적으로 이러한 필요성을 느낀다고 하더라도, 나와 함께 일을 하는 같은 동료들은 이러한 사실을 모를 수도 있습니다.
    - 데이터 엔지니어, 더 나아가서는 데이터 조직은 협업을 하지 않고서는 전사적으로 크나큰 변화를 만들기 어려운 직군이라고 생각합니다. 데이터 엔지니어나 데이터 조직이 일을 잘 하기 위해서는 전사 동료, 혹은 팀들이 어떠한 문제를 가지고 있고, 그 중 데이터를 통해 효과적으로 해결할 수 있는 문제가 무엇인지 선별하여 좋은 해결책을 제시할 수 있어야 한다고 생각합니다.
    - 당연하게도 이런 협업을 잘 하려면 커뮤니케이션 스킬이 뛰어나면 뛰어날 수록 좋습니다. 특히, 데이터 엔지니어나 데이터 조직이 무슨 일을 하는지 잘 모르는 초기 단계에서는 오히려 사내의 세일즈 조직처럼 움직여야 할 수도 있습니다.
  - Time to market을 지키는 능력
    - 데이터 엔지니어를 비롯한 많은 엔지니어들은 종종 아주 어렵고 한 번에 해결하기 어려우며 해결까지 오랜 시간이 걸리는 문제를 푸는 경우가 있습니다. 당연하게도 스타트업일수록 이런 문제를 푸는 시간은 많이 주어지지 않고요. 그렇기에 일을 적절하게 쪼개고, 마일스톤에 맞춰 배포할 수 있는 능력이 필요합니다.
      - 예를 들어 추천 시스템을 만드는 경우 완벽한 추천 시스템을 1년동안 만들어내는 것보다 (물론 1년을 다 사용한다고 해서 완벽하게 만들 수도 없습니다) 한 달, 2주 주기로 시스템을 만들고 유저 피드백 혹은 방법을 바꿔보면서 개선하는 것이 더욱 나은 방법일 것입니다.
      - 특히 시시각각이 중요한 B2C 서비스의 경우 이 능력이 더더욱 중요합니다. 예를 들어, 경쟁사보다 좋은 서비스를 빨리 내놓는 것은 직무를 막론하고 중요한 능력입니다.
  - Time to market을 지키지 않을 능력
    - 하지만 때로는 무리하게, 혹은 프로토타입을 이용해 time to market을 지키는 것보다 시간을 가지고 수행하는 것이 좋을 때도 있습니다.
      - 예를 들어 서비스에서 계속 사용할 데이터 플랫폼을 구축하는데 플랫폼을 구축하는데 오래 걸린다는 단순한 이유만으로 RDB 한 대를 띄워놓고 데이터를 몰아넣는 것보단, 장기적인 관점에서 데이터 플랫폼을 구축하는 것이 더욱 이득이 된다고 판단한다면 이를 설득하여 실행하는 능력도 있어야 합니다.
      - 이를 위해서는 데이터 도메인에 대한 전문성이 있어야 하고, 이를 바탕으로 나와 같이 일하는 동료들을 설득할 수 있어야 합니다.

사실 위에 적은 내용이 비단 데이터 엔지니어에만 해당되는 내용은 아니고, 웬만한 직무를 가지신 분들이 가지면 가질 수록 좋을 능력이긴 합니다. 다만 데이터 엔지니어의 경우 이러한 능력을 지니는 것이 다른 직무에 비해 더더욱 중요하다고 생각하는데, 왜냐하면 본인의 의지에 따라, 혹은 능력 여하에 따라 회사에 단순히 데이터 플랫폼만 구축하고 운영하는 데이터 엔지니어가 될 수도 있고, 전사적으로 영향을 줄 수 있는 데이터 서비스를 제공하는 데이터 엔지니어가 될 수도 있기 때문입니다. (물론 요즘에는 데이터 엔지니어도 포지션이 점점 세부적으로 변해가고 있기 때문에, 이러한 예는 데이터 문화가 잘 조성되지 않은 작은 스타트업에 적합하다는 사실도 같이 말씀드리고 싶습니다)

위에 적은 내용만 보면 모든 회사에 데이터 엔지니어가 필수적으로 있어야 할 것 같은데요. 하지만 아쉽게도 모든 회사가 데이터 엔지니어를 필요로 하진 않습니다. (혹은, 적어도 중요한 포지션으로 채용할 필요는 없습니다)

어떤 회사가 데이터 엔지니어를 중요하게 채용할까요? 제 생각은 다음과 같습니다.

### 데이터 엔지니어가 활약할 수 있는 회사
- 고객이 명확히 존재하는 회사 (주로 B2C 회사)
  - 고객이 존재하는 회사는 항상 서비스에 진입하는 고객의 모든 데이터를 궁금해합니다. 그리고 고객이 발생시키는 모든 데이터를 궁금해하고요. 데이터 엔지니어를 비롯한 데이터 조직이 가장 신나게 활약할 수 있는 회사입니다.
- 데이터 기술 플랫폼을 만드는 회사
  - 다양한 목적으로 데이터 기술 플랫폼을 만드는 회사들이 의외로 한국에도 있습니다. 특히, 대기업의 경우 사내에서 사용하는 도구, 기술, 프레임워크들이 꽤나 많고 비중있게 유지보수되기도 합니다. 이때, 도메인 지식을 가진 데이터 엔지니어들이 활약할 수 있는 부분이 많다고 생각합니다.

다만 위 두 회사가 아니라면, 데이터 엔지니어를 비롯한 데이터 직군이 활약할 여지가 상대적으로 적다고 생각합니다. 데이터는 주의깊게 봐줄 사람이 있을 때만 의미가 있는 거니까요.

지금까지는 스타트업 데이터 엔지니어의 역할에 대해 살펴봤습니다. 그렇다면 좀 더 규모가 큰 회사의 데이터 엔지니어는 어떤 일을 하고 있을까요? (위에서도 그랬지만 여기서부터는 더욱 더 순수한 저의 순수한 생각입니다)

  - 비유하자면, 스타트업 데이터 엔지니어가 generalist에 가깝게 일을 해야한다면, 대기업 데이터 엔지니어는 조금 더 specialist에 가깝게 일을 해야한다는 느낌을 받았습니다.
  - 그래서 사용하는 기술의 숫자가 적더라도 좀 더 깊게 봐야한다던가, 그리고 스타트업에 비해 좀 더 긴 호흡으로 기술을 고도화하는 경우가 스타트업에 비해 더 많을 것 같다는 생각입니다. 그도 그럴 것이, 스타트업에서는 그런 기술 고도화가 필요하지 않을 수 있습니다. 예를 들어 우리나라에서 네이버, 카카오만큼의 트래픽과 데이터를 소화하는 곳은 많지 않고, 그렇게 더욱 큰 데이터를 처리하는 곳에서는 좀 더 튼튼한 기술들이 필요할 테니까요.
