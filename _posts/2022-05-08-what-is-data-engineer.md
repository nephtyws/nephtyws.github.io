> 혹시 코딩 할 줄 아세요?

2019년에 이직했을 당시 회사의 개발자 동료에게 들어본 말입니다. 당시 저는 적당한 규모의 스타트업의 첫 번째 데이터 엔지니어로 입사했었는데, 아무래도 데이터 엔지니어라는 포지션이 생소하고 어떤 일을 하는지 잘 알려져 있지 않아서 그런 것 같다고 생각했지만..

3년이 흐른 지금도 데이터 엔지니어라는 포지션의 정의, 그리고 도대체 회사에서 어떤 일을 하는지에 대한 것들이 다른 직군 분들이 보시기에 정확히 정리되진 않은 것 같습니다. (심지어는 데이터 엔지니어들 사이에서도요!) 최근에 블라인드 게시글 중 `데이터 엔지니어를 개발자로 취급해야 하나요?` 라는 글도 봤었고, 하물며 저한테 `데이터 엔지니어는 무슨 일을 하나요?` 하고 물어봐도 한 문장으로 설명하기가 어렵기도 하고요. 그래서 이 기회에 요즘 세상의 데이터 엔지니어란 무엇이고, 어떤 일을 하며, 최근의 트렌드가 무엇인지 한 번 정리해보는 시간을 가져보려 합니다.

채용 공고만 보고 감이 오시나요? 공통적으로 등장하는 항목들을 조금 정리해보면,
- 서비스의 다양한 곳에서 발생하는 정형/비정형 데이터를 수집하기 위한 배치/실시간 파이프라인 구축 및 운영
- 데이터 레이크/데이터 웨어하우스 등 전사 인원들의 원활한 데이터 분석을 위한 데이터 플랫폼 운영
- 대용량의 데이터 가공을 통해 실제 서비스 혹은 분석에 사용될 수 있는 데이터 마트 구축
- 데이터 기반 서비스 제공 및 서빙 - 관련 `*Ops` 또한 담당

이런 내용들이 공통적으로 등장하는 것을 확인할 수 있습니다. 이제 데이터 엔지니어의 입장에서 이러한 이야기들을 조금 더 풀어서 써보면 다음과 같이 정리할 수 있겠네요.

#### 데이터 엔지니어의 업무
- **서비스의 다양한 곳에서 발생하는 정형/비정형 데이터를 수집하기 위한 배치/실시간 파이프라인 구축 및 운영**
  - 웹, 앱 등의 클라이언트 환경, DB나 서버같은 내부 백엔드 환경, 그리고 third-party tool (주로 마케팅 툴 - Facebook Ad, Appsflyer, ...) 에서 발생하는 데이터들을 ETL/ELT
    - 이를 위해 batch/streaming pipeline을 구성하고, data schema를 정의하고, ETL/ELT를 위해 분산 컴퓨팅 프레임워크 (Spark, Presto, Athena, BigQuery) 를 사용합니다.
- **데이터 레이크/데이터 웨어하우스 등 전사 인원들의 원활한 데이터 분석을 위한 데이터 플랫폼 운영**
  - 클라우드 바깥이라면 주로 Hadoop + ecosystem 같은 기술들, 그리고 클라우드 안이라면 Athena, BigQuery를 사용하는 것이 현재 트렌드입니다.
    - Snowflake, Databricks 등의 상용 SaaS 플레이어들도 존재하고 있습니다.
- **대용량의 데이터 가공을 통해 실제 서비스 혹은 분석에 사용될 수 있는 데이터 마트 구축**
  - 전사 고객들이 데이터에 원활하게 접근할 수 있어야 하기 때문에, 원천 데이터에서 다양한 추상화를 거쳐 주로 데이터 마트라고 부르는 가공 데이터를 만들기도 하고, 조회를 위한 BI tool (Tableau, Redash, Superset, Jupyter, Zeppelin, ...) 도 운영하며, 요새는 누구나 ad-hoc하게 분석할 수 있게 해주는 Amplitude와 같은 tool도 운영하곤 합니다.
  - 또한 이러한 데이터들은 요즈음에는 단순 분석용으로만 사용되는 것이 아니고, 결제/정산 데이터처럼 추가적인 백엔드 원천 데이터로 사용되기도 하고, 고객에게 서빙하기 위해 만들어내는 데이터도 존재합니다. (쿠폰/멤버십/검색/추천 등 데이터를 기반으로 만들 수 있는 다양한 서비스들)
- **데이터 기반 서비스 제공 및 서빙 - 관련 `*Ops` 또한 담당**
  - 위에 언급된 일을 하기 위해 보통 사내에 데이터 조직이 따로 존재하고, 해당 조직의 데이터 엔지니어들이 데이터 관련 서비스의 DevOps (배포, 모니터링 등) 를 주된 role로 가져가는 경우가 많습니다.
    - DevOps 엔지니어분들은 항상 바쁘시기도 하시고, 데이터 환경 자체가 production 환경과는 격리된 경우도 있고, 또한 데이터 관련 배포는 실제 서비스의 배포와는 다른 주기로 이뤄지는 경우가 많기 때문입니다.
    - 요새는 DataOps, MLOps 같은 것들이 부상하면서 이러한 것도 데이터 엔지니어들이 같이 수행하기도 합니다.
  - 또한 AB 테스트, 추천 결과 서빙 등 데이터 기반 서비스를 production으로 내보낼 때 주도적인 역할을 하기도 합니다.

업무를 정리해보니 생각보다 길게 쓰게 되었는데요. 그러면 이러한 업무를 수행하는 데이터 엔지니어들이 가져야 할 기술들은 무엇이 있을까요?

#### 데이터 엔지니어가 갖춰야 할 기술
- Tech skills
  - 분산 처리에 대한 이해
    - 대용량의 데이터를 다루는 만큼 모든 데이터 처리가 분산/병렬로 이루어지는 경우가 많습니다. 물론 high-level framework인 Spark, Athena (Presto), BigQuery, Hive 등이 존재해서 아주 많은 부분이 추상화되긴 했지만, 기본적으로 이런 분산 컴퓨팅 엔진들이 어떻게 작동하는지 알면 좋은 점이 생각보다 많이 발생합니다. 당연하게도 데이터가 아주 많은 환경에서는 생각보다 코드 한 줄 한 줄이 끼치는 영향이 지대할 수밖에 없기 때문입니다.
  - DB/SQL에 대한 이해
    - 데이터 엔지니어가 뛰노는 환경들 (Athena, BigQuery 등) 은 일종의 데이터베이스입니다. 일반적으로 서비스에 활용되는 RDBMS와 다른 부분도 많고, 공유하는 부분도 많습니다. 그렇기에 DB에 대한 일반적인 상식을 많이 알면 알 수록, 특히 위에서 언급한 직접 운용하고 있는 데이터 플랫폼의 기반을 알면 알 수록 운영하기 쉬워지고, 돌발 상황에 대처하기 좋습니다.
    - 또한 요새는 정말 많은 경우에 데이터 조회를 SQL로 하기 때문에, SQL를 잘 짤 수 있어야 하며 일반적으로 서비스에서 사용되지 않는 다양한 집계 함수등의 문법에도 익숙해야합니다.
  - 프로그래밍
    - 당연하게도 이런 업무를 하려면 최소 한 개 이상의 프로그래밍 언어에 익숙해야하며, 데이터 엔지니어는 다른 엔지니어보다 polyglot이 되는 경우가 많습니다.
      - 예를 들어 Airflow와 같은 batch 작업 운용은 Python으로
      - Spark와 같은 분산 처리 프레임워크 활용은 Scala로
      - 밥먹듯이 하는 데이터 조회는 SQL로
      - production 데이터 서빙할 때는 목적에 맞는 API 서버를 만들 수 있는 Go, Javascript 등
  - 클라우드 인프라에 대한 이해
    - 데이터 엔지니어를 위한 DevOps를 전담하는 엔지니어를 채용하는 회사는 없습니다. 그렇기에 혼자 일을 하더라도 클라우드를 잘 활용해야하며, 때로는 DevOps와 관련된 지식들을 많이 익히고 실무에서 활용해야 하는 경우가 많습니다.
      - 클라우드 매니지드 서비스 운영, 배포 파이프라인 구축, Kubernetes 운영, production 환경과 네트워크 연결, 모니터링, IaC 등등..
      - 데이터 엔지니어가 DevOps에 대한 전문가가 될 수는 없지만, 적어도 DevOps 엔지니어 분들과 소통을 통해 문제를 스스로 해결할 수 있을 정도는 되어야 합니다.
- **(특히 스타트업에서 정말 중요한) Soft skills**
  - 서비스에 대한 이해
    - 우리 서비스가 더욱 더 잘 되기 위해서는 어떤 데이터가 필요할지 알아야 하고, 이걸 알아야 데이터를 수집하고 가공하는 등의 행위를 할 수 있습니다.
    - 물론 보통 서비스 전문가는 데이터 분석가나 데이터 사이언티스트, PM 등 데이터 엔지니어보다 더 잘 하시는 분들이 많지만, 그렇다고 해서 데이터 엔지니어가 서비스에 뭐가 중요한지 하나도 모른채 업무를 수행한다면 진정으로 의미있는 데이터를 수집하기 어려울 뿐더러, 미래에 발생할 문제에 대해 예방도 할 수 없습니다. (이때 이런 데이터를 추가로 수집했었더라면 지금 이런 문제 없었을텐데, 그때 schema를 이렇게 정했더라면...)
  - 뛰어난 커뮤니케이션 스킬
    - 조직 내부에서 데이터 엔지니어의 할 일이 잘 정의되어있고, 데이터 엔지니어가 왜 필요한지 알며, 데이터 엔지니어가 어떠한 문제를 해결해줄지 잘 알고 있는 조직은 정말 많지 않습니다. 회사가 그런 필요성을 느낀다고 하더라도, 조직 전체가 그것을 알기는 어렵습니다.
    - 데이터 엔지니어와 데이터 조직은 혼자 덩그러니 있을 때는 회사에 크나큰 변화를 만들기 어려운 직군입니다. 이를 위해서는 stakeholder들이 어떤 문제를 가지고 있고, 그러한 문제를 데이터를 통해 어떻게 해결할 수 있는지 적절히 파악하고, 제시해줄 수 있어야할 때가 많습니다.
    - 이런 것들을 잘 하기 위해 사내에 쌓여있는 중요한 문제들이 어떤 것들이 있고, 그 중 데이터를 활용하면 해결하기 쉬운 문제들이 어떤 것인지 커뮤니케이션을 통해 제시하고, 당연하게도 이후에는 실제로 그 문제를 풀어낼 능력이 있어야 합니다.
  - Time to market을 지키는 능력
    - 데이터 엔지니어를 비롯한 많은 엔지니어들은 종종 아주 어렵고 한 번에 해결하기 어려우며 해결까지 오랜 시간이 걸리는 문제를 해결하는 경우가 있습니다. 당연하게도 그런 문제를 푸는 시간은 많이 주어지지 않고요. 그렇기에 일을 적절하게 쪼개고, 마일스톤에 맞춰 개선할 수 있는 능력이 필요합니다.
    - 예를 들어 추천 시스템을 만드는 경우 완벽한 추천 시스템을 1년동안 만들어내는 것보다 (당연하게도 1년을 사용한다고 해서 완벽하게 만들 수도 없습니다) 한 달, 2주 주기로 모델을 만들고 방법을 바꿔보면서 유저 피드백 혹은 내부 피드백을 기반으로 개선하는 것이 더욱 나은 방법일 것입니다.
    - 특히 시시각각이 중요한 B2C 서비스의 경우 이 능력이 어느정도 있어야 합니다. 회사가 일을 추진하려면 성과가 있어야 하기 때문이죠.
  - 반대로 time to market을 지키지 않을 능력
    - 갑자기 위 이야기와 너무 반대되는 주제긴 하지만, 어떤 주제에서는 time to market보다 시간을 가지고 수행하는 것이 좋을 때도 있습니다.
    - 예를 들어 서비스에서 계속 사용할 데이터 플랫폼을 구축하는데 당장 급하게 운용하기 위해 RDB 한 대를 띄워놓고 데이터를 몰아넣는 것보단, Athena, BigQuery 등의 플랫폼을 각잡고 구축해보는 것이 장기적으로는 좋은 선택이 될 것입니다.
    - 이를 위해서는 데이터 도메인에서의 전문성이 있어야 하고 이를 바탕으로 남들을 설득할 수 있어야 합니다.

데이터 엔지니어도 당연히 소프트웨어 엔지니어의 한 분류이기 때문에, 탄탄한 기술 지식과 프로그래밍 능력은 다다익선입니다. 여기에 더불어 스타트업의 경우는 커뮤니케이션 능력도 정말 중요합니다. 데이터 엔지니어가 커뮤니케이션을 어떻게 하냐에 따라 단순히 분석 환경만 제공해주는 데이터 엔지니어 역할을 할 수도 있고, 혹은 서비스 전반을 아우르는 데이터 서비스 제공자가 될 수도 있기 때문입니다. 그리고 대부분의 회사에서는 후자를 생각하면서 채용을 할 것이기도 하고요.

그러나 모든 회사가 이러한 데이터 엔지니어를 필요로 하는 것은 아닙니다. 예를 들어 B2C가 아니고 B2B 회사라면 데이터 엔지니어의 필요성이 당연하게도 B2C 회사보단 줄어들 것입니다. 또한 회사의 주요 비즈니스 모델이 순수한 기술 기반이라면 (예를 들어 Terraform을 만든 Hashicorp 처럼요) 위와 같은 일을 하는 엔지니어가 필요하지 않을 수도 있습니다. 그래서 데이터 엔지니어라면 자기가 하는 일이 정말 중요하고, 제대로 활용될 수 있는 회사로 가는 것이 중요합니다. 닭 잡는데 소 잡는 칼이 필요하지 않을 수 있으니까요.

지금까지는 스타트업에서의 데이터 엔지니어 역할에 대해 살펴봤습니다. 그렇다면 대기업의 데이터 엔지니어는 어떤 기술을 필요로 할까요? (위에서도 그랬지만 여기서부터는 저의 순수한 생각입니다)

### 데이터 엔지니어의 미래는?
마지막으로는 데이터 엔지니어의 미래에 대해 짧은 제 생각을 몇 자 적어보도록 하려 합니다.

불과 10년 전에는 Hadoop 생태계 속에서 데이터를 처리하는 게 대세였고 (지금도 활발하게 사용되고, 업데이트 되는 기술이죠!) 그리고 최근에는 클라우드의 태동과 함께 데이터 관련 기술들도 많이 성숙해서 Hadoop 이외의 솔루션들도 있고, 클라우드에서 제공하는 managed service들도 있고, Snowflake, Databricks와 데이터 플랫폼의 새로운 지평을 열거나 첨단 기술을 돈 받고 팔 수준으로 끌어올린 사업자들도 많이 있습니다.

그래서 스타트업 수준에서는 이제 인프라나 플랫폼을 걱정하지 않고 데이터를 이용해서 가치를 만들어낼 수 있는 데이터 엔지니어들이 좋은 평가를 받을 것 같습니다. 물론 클라우드를 사용하거나, 상용 서비스를 사용한다 하더라도 실제 세계에는 데이터 엔지니어들이 해소해줘야할 많은 기술적인 문제들이 있지만, 그렇다고 해서 예전처럼 Hadoop cluster를 troubleshooting 하거나 하는 문제들, 혹은 사용하는 어떠한 핵심 기술이 시원찮아서 바퀴를 재발명하는 경우는 많이 없어질 거 같습니다.

그렇다고 해서 데이터 엔지니어에게 기술이 덜 중요해지는 거냐 그런 것은 아닌 거 같고, 여전히 핵심 데이터 기술을 개발할 수 있는 데이터 엔지니어들은 스타트업 말고 어느정도 규모가 있는 회사에서는 중요한 인재가 될 것 같습니다.

다만, 단순히 데이터 플랫폼을 구축하고 이를 유지보수하는 데이터 엔지니어들은 세계적 흐름에 따라 설 자리를 잃을 것 같습니다. 마치 예전에 Hadoop operation 하시던 분들의 일자리가 크게 줄어드는 것처럼요.

데이터적인 사고로 내가 속한 조직이나 회사에 새로운 기회를 열어줄 수 있는 엔지니어, 혹은 날카로운 기술 백그라운드로 데이터 관련 기술에 리더십을 줄 수 있는 엔지니어들이 앞으로의 채용 시장에서도 살아남지 않을까 합니다.
